## Overview

The Message Protocol defines the structured specification for all message types, roles, and communication flows within ClawHive. Messages are the fundamental unit of communication â€” every interaction (user input, LLM response, tool invocation, streaming event) is represented as a message or derived from one.

## Message Object

The core message structure:

```json
{
  "role": "<Role>",
  "content": "<string>",
  "tool_calls": [<ToolCall>, ...],
  "tool_call_id": "<string>"
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `role` | `Role` | Yes | Sender identity: `system`, `user`, `assistant`, or `tool` |
| `content` | `string` | Conditional | Text content. Required for `system`, `user`, `tool`. Optional for `assistant` with `tool_calls` |
| `tool_calls` | `ToolCall[]` | Optional | Present on `assistant` messages that invoke tools |
| `tool_call_id` | `string` | Conditional | Present on `tool` messages. Correlates with `ToolCall.id` |

## Message Roles

### system

Provides instructions and context to the LLM. Not part of conversation history â€” dynamically constructed before each LLM request.

**Construction order:**
1. Soul prompt (agent personality)
2. Notification capabilities description (if configured)
3. Memory context (persisted knowledge entries)
4. Skill prompts (loaded skill instructions)

**Cardinality:** Exactly one per LLM request, at position 0.

### user

Represents input from any external source: human users, cron jobs, other agents, or injected prompts.

| Source | Example | Role |
|--------|---------|------|
| Human input | REPL text entry | `user` |
| Cron job | `[cron:daily_check] Summarize today's activity` | `user` |
| External API | HTTP endpoint message delivery | `user` |

### assistant

Represents LLM-generated output. Can be:
- **Text response**: `content` is set, `tool_calls` is empty
- **Tool invocation**: `tool_calls` is set, LLM is requesting tool execution
- **Mixed**: Both `content` and `tool_calls` present

### tool

Carries tool execution results back to the LLM. Each tool message corresponds to exactly one `ToolCall` from a preceding assistant message.

## ToolCall Object

```json
{
  "id": "call_abc123",
  "name": "exec",
  "arguments": "{\"command\":\"date\"}"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Unique identifier (generated by LLM provider) |
| `name` | string | Tool name, must match a registered tool |
| `arguments` | string | JSON-encoded arguments |

## InputMessage

Extends a user message with metadata about origin and notification routing:

```json
{
  "text": "<string>",
  "source": "<string>",
  "notify": {
    "channels": ["terminal"],
    "priority": "normal"
  }
}
```

| Source Pattern | Description |
|----------------|-------------|
| `user` | Direct human input |
| `cron:<name>` | Scheduled cron job |
| `timer:<id>` | Scheduled notification callback |
| `api:<endpoint>` | External API injection |
| `agent:<name>` | Inter-agent message |

## Streaming Events

Agents emit streaming events through a channel during message processing:

| Event | Description | Key Fields |
|-------|-------------|------------|
| `token` | Text increment from LLM streaming | `token` |
| `triage_decision` | Triage classification result | `triage_decision` |
| `skipped` | Message skipped by triage | â€” |
| `tool_start` | Tool invocation began | `tool_call_id`, `tool_name`, `tool_args` |
| `tool_result` | Tool execution completed | `tool_call_id`, `tool_result` |
| `error` | Error occurred | `error` |
| `done` | Stream complete | â€” |

## Conversation History

- Messages are appended in order as they occur
- System messages are **never** stored in history (reconstructed dynamically)
- History is append-only â€” never modified or reordered
- Persisted as JSONL with timestamps

| Property | Guarantee |
|----------|-----------|
| Durability | Best-effort (failures logged but don't interrupt) |
| Atomicity | One JSONL line per message |
| Completeness | All roles persisted except system |

## Context Window Construction

Before each LLM request:

```
ContextWindow = [SystemMessage] ++ ConversationHistory

SystemMessage = SoulPrompt + NotificationCapabilities? + MemoryContext? + SkillPrompts*
```

- System message is rebuilt on **every** LLM request
- Memory entries loaded from disk on each request (reflects latest saves)
- Skill prompts injected under `## Skill: <name>` headings

## Limits

| Parameter | Value |
|-----------|-------|
| Max tool call rounds | 20 |

| Stream channel buffer | 32 |
